# Design: DID 응급 의료 신원 인증 — 48시간 해커톤 데모 (한국 모바일 신분증 연동)

## Problem Statement

응급 상황에서 환자의 핵심 의료 정보(혈액형, 알레르기, 복용 중인 약, 기저질환, 응급 연락처)를 빠르게 확인할 방법이 없다. 보호자 연락 지연·알레르기 미확인·위험 약물 투여 가능성이 발생한다. 동시에 환자 입장에서는 개인의료정보를 전체 공개하지 않고 응급 상황에 꼭 필요한 정보만 공개할 방법이 없다.

48시간 온사이트 Open DID 계열 해커톤에서, 위 문제를 W3C VC + DID + SD-JWT VC 표준으로 풀되, 한국 모바일 신분증(운전면허증/주민등록증)을 issuer로 내재화 하여 해외 DID 응급 데모와 명확하게 구분되는 30초 데모를 만든다.

## What Makes This Cool

- **30초 안에 "DID 선택적 공개"의 인간 가치를 보여줌**: 환자 QR → 의사가 혈액형·알레르기만 보고, 주민번호·주소는 안 보임. SD-JWT VC가 실제로 동작하는 것을 dev tools에서 확인 가능.
- **한국 모바일 신분증 issuer 연동**: 정부가 서명한 신원 VC가 환자 wallet에 anchor된다. 해외 W3C 데모·OmniOne 자체 홈페이지 어디에도 없는 한국 특화 angle.
- **"환자가 의식 없을 때는?" 아키텍처 슬라이드** (plan-eng-review에서 outside voice 수용 후 다운그레이드): 실제 구현이 아닌 다이어그램으로 시연 — 환자 사전 서명된 authorization policy VC + 의사 verifier role VC + NFC/BLE 트리거 흐름. "프로덕션에서는 이렇게 돌아간다"는 1슬라이드로 설명, 구현 시간은 QR 흐름 + doctor 화면 polish + mDL 배지에 집중.
- **AI lite 자연스러운 통합**: 의사 화면이 알레르기 VC + 처방 예정 약물 → 한국 DUR(의약품안전사용서비스) 데이터셋 기반 정적 룩업으로 빨간 경고. "LLM 확장 가능"은 슬라이드에 한 줄.

## Constraints

- **시간**: 48시간 온사이트. 실제 코딩 가능 시간 약 40-44시간 (수면·발표 준비 제외).
- **인원**: 가정 1인 또는 소규모 팀.
- **타깃**: Open DID Foundation / OmniOne 계열 DID 해커톤 심사위원. DID 철학·기술적 정확성·자사 제품 활용을 봄.
- **기술 표준**: W3C VC Data Model, DID, SD-JWT VC, OpenID4VP 호환. 한국 mDL(모바일 운전면허증) 표준 시뮬.
- **개인정보**: 데모용 가짜 환자 데이터만. 실제 의료 정보 사용 금지.
- **결과물 형태**: PWA (의사 웹 화면 + 환자 모바일 PWA wallet). 네이티브 앱 아님.
- **백엔드**: 없음. presentation 검증은 의사 브라우저에서 직접.

## Premises

1. **응급실(Feature 1) 메인, 마약성(Feature 2) 아키텍처 슬라이드 1장 고정** — 48시간에 두 기능 다 돌리면 둘 다 반쪽.
2. **한국 mDL 표준을 reference한 데모용 issuer(ed25519 키, "Korean mDL Reference Issuer (Demo)" 라벨, "정부"/"행정안전부" 명칭 직접 차용 금지)** 연동이 Feature 1 안에서 유일한 강력한 차별점. 심사위원이 정부 협업 여부를 물으면 reference implementation임을 즉시 명시.
3. **VC 3개(mDL, 혈액형, 알레르기) 모두 실제 SD-JWT 서명·검증**. 빌드 타임 pre-signed. 완전 mockup은 DID 재단 심사에서 자살.
4. **NIMS 비교 제외, E-Gen 비교 슬라이드 제외(결정)** — 시간 우선순위 낮음, 슬라이드 5장 안에 안 들어감.
5. **정적 룩업 (LLM 없음, "AI lite" 라벨)** — 한국 DUR 데이터 기반 알레르기-약물 충돌 매트릭스. 발표 시 "확장 시 LLM" 언급만.
6. **단순 SPA** (PWA installable cut by plan-eng-review). 의사 화면(/doctor) + 환자 지갑(/wallet) 같은 origin React Router. **데모 시연 환경은 doctor 페이지 안에 wallet iframe (`<iframe src="/wallet?embed=1">`)으로 임베드**: doctor가 메인 뷰, wallet은 우측 패널 iframe. same-origin postMessage로 통신. 시크릿·프라이빗·다른 프로필에서도 안정. 심사위원이 "내 폰에서 해볼 수 있어요?"에는 Vercel URL 공유로 wallet 단독 체험만 가능, 역방향 호출은 시연 머신 한정.
7. **핵심 데모 흐름은 30초·한 환자·한 의사·QR 단일 흐름**. (D8에서 iframe 역방향 호출 cut, 아키텍처 슬라이드로만.) 30초 분배: 0-10초 wallet 소개 + mDL 한국 angle, 10-25초 QR + DDI 충돌 경고, 25-30초 자유 체험 prompt.
8. **VC는 빌드 타임 pre-signed** — 실시간 OpenID4VCI issuance 흐름 데모 안 함, 슬라이드 1장으로 설명. 정부 issuer + 병원 issuer ed25519 keypair는 repo에 hardcode (데모 키, CI secret 아님).

## Cross-Model Perspective

독립적 Claude subagent 콜드 리딩에서 다음 통찰 채택:

- **"역방향 호출" 패턴**: 환자가 능동적으로 QR을 보여주는 흐름이 응급 상황의 본질(환자 의식없음)과 부합하지 않음을 지적. 사전 서명된 authorization policy VC + verifier role VC로 자동 응답 시나리오 제안. → 하이브리드(QR 메인 + 역방향 온세컨드)로 채택.
- **빌드 타임 pre-signed VC**: 실시간 issuance 흐름은 48시간 데모 가치 대비 시간 낭비. ed25519 keypair hardcode하고 정적 import → 채택.
- **OSS 추천**: Sphereon SSI-SDK는 의존성 무거우니 sd-jwt-js (openwallet-foundation)을 1차 선택, Sphereon은 fallback → 채택.
- **데이터셋**: RxNorm DDI 대신 한국 DUR(의약품안전사용서비스) 공개 데이터셋이 한국 의료 맥락에 자연스럽다 → 채택.
- **6/24/36시간 체크포인트**: 위험 관리 체크포인트 → 채택.

## Approaches Considered

### Approach A: 안전한 미니멈 (Safe Minimal)
did:key 기반, QR 단일 흐름. SD-JWT 핵심 1개 VC만 실제 서명. 역방향 호출·AI lite·마약성 슬라이드 모두 제외.
- 효력: 24-30시간, 위험 낮음.
- 단점: 차별화 1개 포인트(한국 모바일 신분증)뿐. 심사위원이 "역방향 호출은?" 질문 시 못 답함.

### Approach B: 추천 본진 (Recommended Core) ★
하이브리드 QR + 역방향 호출 온세컨드. SD-JWT VC 실제 서명·검증. mDL Reference Issuer 시뮬. 정적 DUR 룩업 (LLM 없음). 마약성 아키텍처 1슬라이드.
- 효력: 40-44시간 코딩 + 4시간 슬라이드/리허설 (수면 6시간 별도), 위험 중간.
- 장점: 30초 데모 두 시나리오, 한국 mDL angle 첫 10초, 기술 깊이 + 이야기 모두.

### Approach C: 프로토콜 쇼케이스 (Protocol Showcase)
OpenID4VCI/VP 전체 프로토콜 흐름 시각화. 네트워크 탭에서 실제 JWT 페이로드를 심사위원에게 시연.
- 효력: 38-42시간, 위험 중-높음.
- 단점: 30초 'whoa' 약함. 의료/공공 섞인 심사위원에게 임팩트 부족.

## Recommended Approach

**Approach B (추천 본진)** — 이야기와 기술 깊이의 교차점. 36시간 체크포인트에서 iframe postMessage 역방향 호출 cut 결정 가능하므로 시간 오버 위험을 능동적으로 제어할 수 있음.

### 30초 데모 시나리오 (확정 — D8에서 역방향 호출 cut, QR 단일 흐름)

**시연 환경**: 단일 노트북, **wallet과 doctor가 같은 origin으로 라우팅** (`/wallet`, `/doctor` React Router). 화면 분할 좌(wallet 모바일 뷰포트 에뮬레이션) + 우(doctor 데스크탑). 시연자가 양쪽 조작.

**카메라 setup**: 외장 모바일 폰을 wallet으로 사용 + 노트북 웹캠으로 폰 QR scan **(권장, 사전 준비 단계에서 확정)**. 가능하지 않으면 fallback: wallet의 QR 캔버스 표시 + `navigator.clipboard.writeText(vp)` 자동 복사 → doctor의 `<textarea>` paste. 심사위원에게 setup 정직하게 명시.

- **0-10초**: wallet 진입 → mDL VC + 혈액형 VC + 알레르기 VC 카드 3개 표시. **mDL 카드에 "🇰🇷 Korean mDL Reference Issuer (Demo)" 배지 + 작게 "(reference implementation, not gov-signed)" disclaimer 표시**. 한국 mDL 차별화 angle 도입.
- **10-20초**: 응급실 도착 → "응급 모드" 토글 → QR 표시 → doctor에서 스캔 → **doctor 화면 상단 "✓ Korean mDL Reference Issuer 서명 검증됨" 배지** + 본문 "혈액형 A+, Rh+, 페니실린 알레르기, 중증" 표시, **이름·주민번호·주소는 마스킹** (●●●). dev tools 한 번 까서 SD-JWT VC 페이로드(`~`로 disclosure 구분) + KB-JWT 구조 확인 가능.
- **20-28초**: 의사가 처방 예정 약물 "Amoxicillin" 선택 → 정적 DUR 룩업 → **빨간 경고: "페니실린계 알레르기 환자 — 충돌"**. 30초 데모의 의료 가치 정점. 미등록 약물(예: Vancomycin) 클릭 시 회색 "수동 확인 필요" 배지.
- **28-30초**: "환자가 의식 없으면?" 1슬라이드 오버레이 — 아키텍처 다이어그램: authorization policy VC + verifier role VC + NFC/BLE 트리거. "프로덕션 구현"으로 명시.

### 기술 스택 (확정)

| 영역 | 1차 선택 | Fallback / 비고 |
|---|---|---|
| 프레임워크 | Vite + React + TypeScript | - |
| SD-JWT VC | `sd-jwt-js` (openwallet-foundation) | `did-jwt-vc` → `@sphereon/ssi-sdk` 순으로 fallback |
| QR 스캔 | `html5-qrcode` | 실패 시 텍스트 페이스트 input 1개 (`<textarea>` SD-JWT VP 직접 붙여넣기) |
| 스타일링 | Tailwind | - |
| DID 메서드 | `did:key` (ed25519 hardcode keypair) — 확정 | - |
| 통신 (역방향) | iframe + window.postMessage (same-origin 검증) | - |
| 정적 룩업 데이터 | DUR(의약품안전사용서비스) 추출 JSON | 직접 5종×10종 매트릭스 손으로 만듦 (1시간 백업) |
| PWA | manifest.json + 최소 service worker | **시간 부족 시 cut 1순위** (Lighthouse 점수는 success criteria 아님) |

### Presentation 검증 정책

- **Trust anchors**: mDL Reference Issuer pubkey + Demo Hospital Issuer pubkey 2개를 `src/data/trust-anchors.ts`로 분리. **wallet과 doctor 양쪽 모두 이 동일 trust anchors를 import**해서 사용 (wallet은 의사 verifier role VC 검증에 사용).
- **Holder key**: 빌드 타임에 고정된 demo holder ed25519 keypair (D2 통합으로 `src/data/issuer-keys.ts`에 함께 export). SD-JWT VC 발급 시 `cnf` 클레임에 holder pubkey 포함. 이론적 흐름(첫 부트 생성 → localStorage)은 슬라이드 1줄로만 설명.
- **Nonce 처리 — QR flow** (D8/D9 후 단일 흐름, randomized): wallet이 QR 생성 시 `crypto.getRandomValues(new Uint8Array(16))`로 매번 random nonce 생성, SD-JWT VP의 KB-JWT 안에 포함. doctor가 VP 파싱 후 nonce 확인. (one-way QR에서 replay 방지에는 부족하지만 dev tools 까는 심사원에게 randomized 값 노출 — 인상 결정성에 중요). 슬라이드에 "프로덕션 single-shot QR replay 방지는 OpenID4VP request_uri" 1줄 명시.
- **Audience**: KB-JWT `aud`는 fixed demo session ID `"hackathon-demo-2026"`. 슬라이드에 "프로덕션은 verifier DID로 audience binding" 1줄.
- **만료**: VP token `exp` 5분, 검증 시 확인.
- **Revocation**: 데모에서 스킵. 슬라이드에 "프로덕션: StatusList 2021" 1줄.

### 역방향 호출 아키텍처 슬라이드 (D8: 구현 cut, 슬라이드만)

다이어그램 슬라이드 1장에 다음 흐름 묘사 — 실제 구현 안 함:

```
[의사 디바이스]                                 [환자 wallet (의식없음)]
    │                                                  │
    │  ── NFC tap or BLE proximity ──►                │
    │   verifier role VC + nonce                       │
    │                                                  │
    │                              wallet receives ◄───┤
    │                              authorization policy VC
    │                              (사전 서명, holder key)
    │                              match: policy.allow ⊇ verifier.role?
    │                              YES → policy.disclose 만 공개
    │                                                  │
    │  ◄── SD-JWT VP (selective) ───────────────────  │
    │   nonce echo, holder binding                     │
    ▼
  의료진 화면 표시
```

"프로덕션 구현 흐름. 본 데모는 30초 시연 시간 제약으로 QR 단방향에 집중, 위 흐름은 W3C VC + DID 표준으로 정합 가능함을 다이어그램으로 시연." 슬라이드 캡션.

### Authorization policy VC (D8 후 슬라이드 다이어그램만)

- 본 데모 코드에는 구현 안 함. 아키텍처 슬라이드에 다음 묘사:
- **Issuer**: 환자 self-issued (holder key 서명, 또는 더 단순한 typed config).
- **Claims**: `{ id, allow: ["emergency_physician"], disclose: ["bloodType", "rhFactor", "allergens", "severity"], validUntil }`.
- 슬라이드 1줄: "프로덕션 wallet은 verifier role VC를 받으면 자동 매칭·응답하는 policy를 사전에 갖고 있음".

### 에러 핸들링 (plan-eng-review 추가, D8 후 axn 갱신)

- **DUR 룩업 미등록 약물 (D5)**: `DDICheck`가 환자 알레르기와 약물 매칭에서 매칭 없을 때, "✅ 충돌 없음" 대신 회색 배지 "⚠️ 데모 룩업에 등록 안 된 약물 — 수동 확인 필요" 표시. silent "안전" 인상 방지.
- **SD-JWT 검증 실패**: doctor의 `verify` 함수가 issuer signature/aud/exp 중 하나 실패하면 빨간 박스 "❌ VC 검증 실패: [reason]" 표시. dev tools 까서 reason 확인 가능 (심사 시연용).
- **Trust anchor 미일치**: issuer DID가 TRUST_ANCHORS에 없으면 "❌ VC 검증 실패: untrusted issuer" 표시.
- **QR 인식 실패**: html5-qrcode 인식 실패 시 textarea paste input으로 자동 노출. 같은 검증 흐름.
- **localStorage seed 누락**: wallet 첫 진입에서 `patient-vcs.ts`의 빌드 타임 VC seed가 안 보이면 (인코그니토 첫 진입 등), 즉시 seed 후 진행. 별도 UI 없음.

(D4 postMessage timeout · `event.origin` 핸들링은 D8 cut과 함께 제거.)

### 응급 모드 토글 정의

- **OFF**: wallet 평시 상태, VC 카드 표시만, QR 생성 불가.
- **ON**: QR 생성 버튼 활성화. QR에 포함되는 disclosure 정책 = `{ mDL: [], bloodType: ["bloodType","rhFactor"], allergy: ["allergens","severity"] }` (mDL 클레임은 0개 — 전체 마스킹, 단 issuer 서명은 검증됨으로써 신원 anchor 효과만 표현).

### Verifier (의사) role VC (D8 후 슬라이드 다이어그램만)

- 본 데모 코드에는 구현 안 함. 아키텍처 슬라이드에 다음 묘사:
- Issuer: 병원 issuer ("Demo Hospital Issuer")
- Holder: 의사 DID (did:key)
- Claims: `{ role: "emergency_physician", hospital: "Demo General Hospital", validFrom, validUntil }`
- 슬라이드 1줄: "프로덕션에서 의사가 NFC/BLE로 wallet에 자기 role VC를 제시하면 policy 매칭으로 자동 응답".

### Patient VC 스키마 (요약)

| VC | Issuer | 전체 클레임 | 응급실 disclosure |
|---|---|---|---|
| mDL VC | Korean mDL Reference Issuer (Demo) | name, residentNumber, birthDate, address, photo, licenseClass | **전체 마스킹 (0개 클레임 공개)** — issuer 서명 검증만으로 신원 anchor 효과 |
| 혈액형 VC | Demo Hospital Issuer | name, bloodType, rhFactor | **bloodType, rhFactor만** |
| 알레르기 VC | Demo Hospital Issuer | name, allergens[], severity[], lastUpdated | **allergens, severity만** |

의사 화면에서 마스킹된 클레임은 SD-JWT undisclosed digest를 `●●●` placeholder로 시각화 (실제 데이터는 wallet에 남아있고, presentation에 포함되지 않음).

### QR 페이로드 인코딩

- SD-JWT VP를 base64url로 직접 raw encoding하여 QR에 담음. 백엔드/서버 reference URL 없음.
- 1-3KB 가능. QR ECC level L 사용, version 자동.
- 인식 실패 fallback: 의사 화면에 `<textarea>` paste input 항상 표시 (작지만 visible). 페이스트 → 같은 검증 흐름.

| localStorage 없음 | incognito 첫 진입 | NO | 즉시 seed | 없음 (사용자 모름) |
| 번들 5MB 초과 | 첫 로드 10초+ | 사전 smoke 측정 | did-jwt-vc fallback | 사전 인지 가능 |

**Critical gap 평가**: 모든 실패 모드가 (a) 테스트로 가려져 있거나, (b) 에러 핸들링 있거나, (c) UI 명시 노출됨. **silent fail이 사용자/심사위원 눈에 안 잡히는 critical gap은 없음.**

## Worktree parallelization strategy

본 데모는 단일 origin의 single SPA. 모든 코드가 `src/` 안에서 서로 호출. 의존성 그래프:

```
issuer-keys.ts ←─ patient-vcs.ts ←─ scripts/sign-vcs.ts (빌드 타임)
       ↑                 ↓
  trust-anchors          localStorage seed
       ↑                 ↓
   sdjwt.ts ←──────── WalletScreen.tsx ──┐
       ↑                                  │
       ├── DoctorScreen.tsx ←──────── (clipboard or QR)
       └── DDICheck.tsx ←── dur-lookup.json
```

**Sequential implementation, no parallelization opportunity.** 1인 개발자 48시간이고 모든 모듈이 sdjwt.ts에 수렴. worktree 병렬화는 의미 없음.

## What I noticed about how you think

- **"수정된 전제 모두 수용, Phase 4로 진행"** — 제가 강하게 push한 P2(실제 VC 서명 필수)에 동의했지만, P1·P3·P4는 본인 의견으로 수정을 요구. "다 받아들이는" 사람이 아니라 "필요한 곳에서 양보하는" 사람의 패턴. 해커톤 48시간 동안 트레이드오프 결정을 빠르게 내리는 데 중요한 기질.
- **"둘 다 얻는 하이브리드 데모 (A 완전히, B는 모드 전환 화면만)"** — D4에서 추천한 단일 기능 집중을 거부하고 더 야심찬 하이브리드를 선택. D11에서도 안전한 A 대신 추천 B를 선택. 일관되게 "야심 + 절제"의 중간을 고름. 너무 안전하면 차별화 부족, 너무 야심차면 안 끝남 — 본인은 그 균형점을 본능적으로 인지하고 있음.
